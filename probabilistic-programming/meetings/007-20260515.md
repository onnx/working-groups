## Summary

### Stan Architecture Evolution Discussion
Brian and Adam discussed the evolution of Stan’s architecture, including CmdStan, BridgeStan, and the possibility of future JAX-based backends. The conversation also touched on PyMC’s planned Rust backend efforts and the broader challenges of maintaining computational backends for probabilistic programming systems. Brian noted plans to continue restructuring the RNG documentation.

### Random Number Generator Testing Methods
The team discussed modern testing approaches for random number generation systems. Adam explained that current ONNX implementations rely primarily on NumPy and the C++ standard library without validating statistical equivalence against frameworks like JAX. The group agreed that rigorous testing methodologies would become increasingly important as ONNX introduces its own RNG implementations.

### ONNX Random Number Generation Implementation
Adam outlined the proposed implementation path for RNG functionality within ONNX, beginning with basic primitives such as uniform doubles and integer sampling before layering higher-level distributions like Gaussian, Dirichlet, and binomial sampling. Brian suggested consulting Bob Carpenter to better understand Stan’s RNG testing strategies and implementation practices.

### Hash Operator Implementation Simplification
Adam and Brian discussed simplifying operator implementations by using hexadecimal serialization of 128-bit state values rather than introducing additional hashing operators. The group agreed that this approach would simplify serialization while remaining sufficiently compact and deterministic for RNG state management.

### Random Number Generation Specifications Refinement
The discussion focused heavily on narrowing the scope of the RNG proposal. The group agreed to concentrate on INIT, SPLIT, and modifications to existing uniform and normal operations while deferring additional distribution-specific operators to future proposals. A separate follow-up document will eventually cover distributions such as categorical, Bernoulli, Beta, and Dirichlet sampling.

### RNG Document Focus Areas
The team aligned on restructuring the RNG specification into a concise document focused on the three primary operator modifications. Brian agreed to produce a cleaned-up draft for Slack review. Andreas noted that ONNX’s RFC template may itself require revisions and suggested the possibility of contributing improvements to the RFC process.

### ONNX Random Number Generator Implementation
The group reviewed ONNX Runtime’s current RNG implementation strategy and discussed testing compatibility with JAX. Adam explained that ONNX Runtime currently relies on standard library RNG behavior, while the reference implementation uses NumPy. Andreas confirmed that JAX could be added as a test dependency without becoming a permanent build dependency.

### ONNX Meetup Presentation Planning
The team discussed the upcoming ONNX meetup hosted at Microsoft in California on June 26th. Andreas shared that working groups would have opportunities to present progress either live or through pre-recorded presentations. Brian confirmed interest in participating and requested the submission information be shared in Slack.

### ONNX and AI Frameworks Discussion
The meeting concluded with broader discussion around the current state of AI frameworks and ONNX ecosystem activity. Andreas noted that recent ONNX development has focused more on maintenance and security-related tasks than large feature additions. The team also discussed PyTorch’s continued dominance in industry and JAX’s growing popularity in research and probabilistic programming workflows.

### Discussion Topic: ONNX RNG Operator Specification Review

The group reviewed the current ONNX random number generation operator proposals.

Adam confirmed that a separate `num_outputs` attribute is unnecessary because the graph structure already determines the number of outputs produced by the SplitPRNG operator.

Additional cleanup items were identified, including:

* Removal of duplicated sections within the specification
* Clarification of operator semantics
* Consolidation of seed-related documentation

The group agreed to continue refining the specification before broader review.

---

### Discussion Topic: SplitPRNG Optional Data Input

The discussion focused on the optional `data` input proposed for the SplitPRNG operator.

Adam questioned whether the semantics were sufficiently constrained and suggested a stricter definition:

* Input should be rank-1
* Number of elements should equal the number of outputs
* Each element should uniquely map to one output stream
* Output identifiers should be unique

The group discussed how similar approaches have been used in other systems and agreed that further specification would be required if the feature remains part of the proposal.

---

### Discussion Topic: RNG Splitting and Reproducibility

Adam and Brian reviewed the value of RNG splitting for:

* Parallel computation
* Distributed execution
* Reproducible workflows
* MCMC execution across multiple chains

While the feature offers benefits for structured random number generation, Adam questioned whether it provides significant advantages beyond simple integer-based indexing.

The group agreed that any implementation should guarantee unique derived states for unique identifiers.

No final decision was made regarding inclusion of the optional data input mechanism.

---

### Discussion Topic: PRNG Algorithm Selection

The group discussed whether ONNX should:

1. Mandate a specific PRNG algorithm
2. Allow algorithm selection through an operator attribute

Brian proposed using the Threefry PRNG implementation from JAX due to its maturity and extensive testing.

Adam suggested discussing the proposal with the ONNX Operators Working Group before making a final decision.

The group also considered an extensible design similar to Java's Random framework, where different algorithms could potentially be supported in the future.

---

### Discussion Topic: ONNX Change Proposal Strategy

Adam and Brian discussed how to present the RNG proposal to the ONNX community.

The group agreed that:

* The proposal should emphasize practical benefits and existing precedent.
* Feedback from operators and runtime implementers will be important.
* Future use cases should remain possible without overcomplicating the initial design.

A future write-up will explain the rationale for selecting Threefry and discuss tradeoffs between fixed and configurable PRNG algorithms.

---

### Discussion Topic: RandomUniform and RandomNormal Operator Updates

The group reviewed required updates to existing ONNX random operators.

Proposed changes include:

* Allowing `int64` outputs for RandomUniform.
* Defining `low` and `high` parameters as type `T` instead of restricting them to floats.
* Making corresponding changes to RandomUniformLike where appropriate.

These changes would make the operators more useful as building blocks for additional sampling algorithms without requiring separate random bit generation operators.

The group also discussed dynamic-shape behavior and the role of "uniform-like" operators in static computational graphs.

---

### Discussion Topic: Reproducibility Across Frameworks

Adam and Brian discussed reproducibility considerations across frameworks such as:

* JAX
* Stan
* PyMC

The group acknowledged that exact bitwise reproducibility across frameworks may be difficult due to implementation differences.

The current goal is deterministic behavior within ONNX runtimes rather than guaranteeing identical outputs across all external frameworks.
